Executive Summary

The Problem:

Our project is based on the original design of the CS3319 final project that involved the creation of a bus trip booking system with a functional database. This previous design suffers from a lackluster front-end usability and a database-centered design. In our groups testing, this design proved to implement no design principles clearly, along with a confusing layout and difficult to read information.

Highlighted issues begin with the absence of any external consistency between this design and other similar trip booking services in the market. Missing features of wishlist, login, signup, and home pages as well as aesthetics contribute to this issue. Additionally, the user's working memory is challenged by the failure to properly use the visibility and affordance principles. Buttons are all laid out on a single page with no feedback apart from textual descriptions to indicate their clickable nature. Therefore, by making no use of the visibility or affordance principles, it forces the user to use more of their working memory towards the functionality of the website over their own goals. Finally, the original design lacked flexibility and efficiency as each piece of functionality only had a single avenue to perform the task with each avenue being either confusing or too complex.

Overall, the design of the original CS3319 bus trip project suffered from a poor user-centered design, making the project feel unusable from a user experience perspective. The goal of our group project is to redesign the entire user experience from the ground up using the design principles and methodologies learned within this course.

Our Approach:

To create a more usable and effective design, our group aims to create a better user-centered design through the implementation of appropriate design principles.

Plan:

- Adding the new web pages of a sign in, sign up, home, and wishlist page.

- Extensive user tasking

- Reorder all web pages

- Divide web pages to group members

- On each page, identify the key issues and areas for improvement

- Create mock Figma redesigns of our web pages with newly integrated design principles

- Group critiques web pages

- Final HTML implementation of web pages

- Heuristic evaluation

Results:

With the addition of new web pages, we were able to ensure that our redesign was externally consistent with the features found within the industry standard. Continuing this consistency, our user tasking allowed us to more effectively reorder all the webpages into a layout that accounted for our desired user flows. On top of this, the reordering aided the redesign in resolving the visibility, working memory, flexibility and efficiency issues with the original design. By individually working on separate web pages, our team effectively created unique design solutions that were made consistent across the system through the group critiques. Each web page became the host of a variety of design principles with things like affordance through button colouring, error prevention, and metaphors as some of the highlights. By evaluating our redesign with Nielsonâ€™s 10 heuristics for design, our group confirmed the success of the system as it featured each heuristic throughout.

Conclusion:

In conclusion, our redesign of the CS3319 bus trip booking system is effective at creating a more user-centered design that is intuitive and easy to use. The redesign shows our implementations of a number of design principles that improve upon the function and usability of the system.

Design Princples

Affordance:

We used affordance across the system with our button shading artifact that ensures the user can easily tell whether a button is clickable or not. Notable examples are in the signup page that highlights fields or buttons in use as well as the trip information page that shades the arrow buttons when hovering over them for scrolling through photos.

Fitts' Law:

Found within the signup and login pages, the subtext for creating an account or requesting a password reset is directly below the respective signup and password fields, putting it at an easily reachable distance.

Constraints

Constraints are found throughout the payment pages where information about the selected trip is auto populated, thus reducing the amount of actions the user can undertake. On top of this, the payment page only allows for critical information such as credit card numbers and postal codes to be inputted in one format.

Progressive Disclosure

On the account creation page, progressive disclosure is used to separate information and settings options for the user's account across several different tabs; overview, payment information, past trips, and settings. This is to make the information more palatable by releasing it in smaller chunks for the user to engage with.

For the booking pages, this is represented in an accordion form that will only display one tab at a time to avoid overwhelming the user.

Interactivity

On the wishlist page, the user can drag and drop their wishlisted trips into different positions with their mouse cursor, this is an example of the manipulation form of interactive design principles as the user can physically change the position of the wishlisted items.

Metaphors

In our project, metaphors are generally used to enable familiarity with a symbol or button. An example of this would be the eye symbol in the password field that reveals and hides the typed characters once clicked, representing the visibility of the password. Another example is the heart icon used on the navigation bar that brings the user to the wishlist page, representing the area where liked trips reside.

Feedback

Throughout our system, it is represented in different forms with the goal of giving the user more information as to what events are taking place. During the signup phase of account creation, the user will be prompted for a password, as the password is typed, the originally red rules will begin to change to green or if ignored will stay red. Additionally, on the login page, if the user has an unsuccessful login, a message will display saying â€˜Incorrect login or passwordâ€™ to keep the user informed on why it was unsuccessful.

80/20 Rule

For the account information page, we enable the 80/20 rule by placing tabs with extended contextual functionality for that tab. For example, features in the settings tab like â€˜Delete accountâ€™ are found only within that tab. Therefore on each tab, there is 20% visible functionality, with the rest being layered or in menus. This is effective since the â€˜Overviewâ€™ tab shows the most relevant or popular features with the other more in depth features hidden in the layers of the webpage.

Flexibility

Our system approaches flexibility by offering a range of options for the user under the account settings page. Some settings options include the ability to enable or disable notifications about trip sales and allowing or disallowing the receipt of relevant emails from our travel company. These settings allow the user to customize their experience with the system in various ways to suit their needs or wants.

Mapping

When booking a trip, the mapping design principle allowed us to define the order of the actions in an intuitive and easy to understand way for the user. It makes more sense to display the trip and passenger information to the user first so that there is no confusion or selections needed before proceeding to purchasing. This is how our system functions, first the user will be shown the trip information being selected, then they can enter their passenger information and confirm and only after this confirmation is the user brought to the card and billing information pages.

Confirmation

To remove wishlisted items, the user must first click the â€˜Edit Wishlistâ€™ button to enable editing features, once clicked this button brings up a confirmation screen to ensure that the user is meant to enter editing mode and if not, a way to go back to the original screen.

Confirmation is also represented on the booking pages that require user confirmation before proceeding to payment, and to confirm the payment information is correct.

Diversity

The wishlist page features a ranking system based on the placement of wishlisted trips, defaulting to the order in which the trips were wishlisted. If the user were wanting to rearrange these items to show more interesting ones at the top, there are two separate functions to allow this. First, the user can click on the trip card and drag the item to its desired space. Otherwise, the user can click the up and down arrow buttons to move the item card one spot up or down at a time. Therefore, the diversity comes from the ability to move these trip cards with unique functionalities to choose from.

Selective Attention

Our redesign improves upon the selective attention principle by our initial reorder of all the webpages into a more intuitive layout. Especially in terms of user flow in relation to our defined user goals, the reorder provided a much more user-friendly layout, where all the function of the website appears in the proper context.

Hick's Law

Hickâ€™s Law is represented in our system on the search and results page that reduces the number of trips being shown as the user begins to add additional search criteria. For example, if the full database of trips is being shown to begin with, when the user begins typing a location, the number of resulting trips will decrease until only the relevant trips are in the results section.

Fidelity Principle

Using the fidelity principle, the trip information page only displays information that is relevant to the trip being viewed. On each trip page, the user will be able to find all the information about that trip alone and nothing else to ensure that all the information represented is useful.

Simplicity

When booking a trip, payment information requires the input of credit card information where each field such as number, expiry date, and CVV are formatted differently. To make entering this information simpler to the user, we designed these fields with auto-formatting that automatically takes the userâ€™s entries and places them into the proper formatting of each respective field.

Consistency

Consistency was an element of focus throughout the system in the form of both external and internal consistency. For external, our system implements similar features to those of other travel websites. For internal, our system uses the same navigation bar throughout the entire experience as well as the use of the same colour palette, button styles, and trip card styles across the system.

Learnability

Our system is learnable through not only the aid of tooltips and easy to read error messages that instruct the user in various ways, but also through the help document provided that gives an overview of each webpage in the system and answers some basic Q&A questions like â€˜How do I book a ticket?â€™.

Working Memory Principle

By providing the user autofilled information of the selected trip in the booking pages, the user does not have to remember any extra information about the trip in order to complete a booking. On top of this, the wishlist page provides the user a way to reduce the need for working memory by allowing them to favourite trips that seem interesting. Then instead of having to search for them again which would require more memory, they can simply navigate to the wishlist page to see the trip again.

Heuristic Evaluations

1. Visibilty of System Status

The first principle is about keeping users informed at each step of their interaction with the website. Therefore, the website should be predictable which means that users should be able to know what to expect in the next interaction. Our bus booking travel agency website is easy to predict at every step of interaction.

For example, the buttons on the main page helps users to expect what the next page will be if they click on those buttons.

Also, the wishlist page is another valuable example to the visibility of the system status. As it can be seen, a user can assume the next step of their action by clicking on the buttons.
